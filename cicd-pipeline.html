The Mission

At Cyberoide Tech, deployments of .NET applications were manual, slow, and error-prone — builds happened on developer machines, configuration lived in people's heads, and a production release was a calendar event.

I built end-to-end CI/CD pipelines that take every commit from push to production automatically: compiled, tested, containerized, and deployed to AWS with the same repeatable process for dev and production.

What I Built

  • GitHub Actions workflows for automated builds, test runs, and artifact publishing on every push and PR.
  • GitLab CI/CD pipelines with stage-gated promotion from dev to production.
  • Multi-stage Jenkins pipelines (at Emumba) with encrypted secrets management baked in.
  • Dockerized build outputs for consistent runtime environments across every stage.
  • Environment-scoped secrets — credentials injected at deploy time, never committed.
  • Deployments wired to AWS (EC2, ECS, RDS) for high availability and rapid recovery.

Challenges & Solutions

"Works on my machine" builds

Builds done locally produced inconsistent artifacts and untracked dependencies.

Solution: All builds moved into pipeline runners with pinned toolchains and Docker images — every artifact reproducible from a commit SHA.

Credentials scattered across scripts and machines

Connection strings and API keys lived in plaintext config files and shell history.

Solution: Centralized encrypted secrets in GitHub/GitLab secret stores with environment-level scoping, injected only at runtime — nothing sensitive in the repo.

Production deploys were high-risk events

Manual multi-step releases meant long change windows and frequent human error.

Solution: Stage-gated pipelines where production deploys reuse the exact artifact validated in dev, reducing releases to an approval click.
