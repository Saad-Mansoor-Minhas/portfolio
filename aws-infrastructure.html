High-Availability AWS Architecture

A production container-hosting stack — ECR → EC2 → ASG → ALB → Route53 — provisioned entirely with Terraform and deployed automatically through Bitbucket Pipelines. Designed for zero-downtime releases and self-healing scale.

The Mission

A client running containerized workloads needed infrastructure that could survive instance failures, absorb traffic spikes, and deploy new versions without downtime — all without a dedicated ops team babysitting it.

The answer: a fully codified AWS stack where Docker images flow from ECR onto EC2 instances managed by an Auto Scaling Group, fronted by an Application Load Balancer with DNS managed in Route53. Every resource defined in Terraform, every deploy triggered by a git push.

What I Built

  • Modular Terraform — reusable modules for networking, compute, and DNS, parameterized per environment.
  • Auto Scaling Group with health-check-driven instance replacement and scaling policies tied to load.
  • ALB with target groups and health checks for graceful rolling instance turnover.
  • Bitbucket Pipelines building images, pushing to ECR, and rolling out via launch template refresh.
  • Route53 records aliased to the ALB for clean, zone-managed DNS.
  • A parallel ECS (EC2 launch type) variant with ALB + ASG for teams preferring managed orchestration.

Challenges & Solutions

Deployments caused brief outages

Replacing instances naively dropped in-flight requests during releases.

Solution: ALB connection draining plus ASG instance refresh with health-check gating — new instances must pass checks before old ones terminate, making deploys zero-downtime.

Configuration drift across environments

Manually-created dev and prod environments kept diverging, causing "works in dev" failures.

Solution: Single set of Terraform modules with per-environment variable files and remote state — identical infrastructure stamped out per environment, reviewed via pull requests.

Auto-scaling misconfigurations in production

Inherited scaling policies thrashed — scaling up and down repeatedly under steady load.

Solution: Re-tuned CloudWatch alarms with proper cooldowns and target-tracking policies based on real traffic patterns, stabilizing the fleet and cutting wasted instance-hours.
