Making risky releases routine on AKS.

A canary release strategy with percentage-based traffic splitting for containerized production workloads on Azure Kubernetes Service. The previous version stays deployed and warm, so rolling back is a traffic switch — seconds, not a redeploy.

Context

NextGen Technologies, production workloads

Stack

AKS · Kubernetes · Azure DevOps · ACR · Azure Monitor

Year

2025 — present

release vN+1 → canary at 10% → observe → promote to 100% · vN kept on standby

The brief

Every release was all-or-nothing

Deploys to containerized production workloads replaced everything at once: a bad release meant every user felt it immediately, and rolling back meant a scramble to rebuild and redeploy the previous version while the incident clock ran.

I introduced canary releases on AKS. Each new version takes a small, configurable slice of live traffic first, and only takes over fully after proving itself against real users — with the outgoing version held warm the entire time.

What I built

The work

  • Percentage-based traffic splitting between stable and canary deployments on AKS.
  • Parallel versioned deployments — the prior release stays scaled and ready for instant rollback.
  • Azure DevOps pipelines driving image builds to ACR and progressive rollout stages.
  • Health-gated promotion — canaries watched through Azure Monitor before taking full traffic.
  • RBAC governance and automated secret rotation across the cluster and surrounding Azure resources.
Problems along the way

Challenges

Full blast radius on day one

Rolling updates replaced every pod regardless of how the new version behaved under real traffic.

Fixed with weighted traffic splitting — new versions serve roughly 10% of requests first, promoted only once error rates and latency hold within thresholds.

Rollbacks were slow and manual

Reverting meant redeploying the previous image — minutes of degraded service every time.

Fixed with the prior version kept deployed and scaled; rollback became an immediate traffic-weight change measured in seconds.

Secrets and access sprawl

Long-lived credentials and broad access made security incidents slow to contain.

Fixed with RBAC governance and automated secret rotation via Key Vault — cutting security-incident response time by 30%.
